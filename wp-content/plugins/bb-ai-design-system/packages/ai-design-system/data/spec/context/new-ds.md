### Creating a New Design System

No site design system exists yet. Generate a complete token set, reset CSS, and base CSS along with your section CSS. The tokens you create will become the site-wide design system for future pages. You may also create base JavaScript (`/* @base */`) for utilities that should be reused across every page on this DS — `onReady`, breakpoint helpers, generic observers. Consider extracting a few commonly reusable primitives into base CSS — buttons, badges, dividers, and containers are good candidates. Keep it to a handful of truly generic patterns; most styling belongs in section CSS.

**`@base` is per-DS; `@page` is per-page.** Anything under `/* @base */` (CSS or JS) becomes part of the design system itself and runs on every future page that uses it — that's the right home for any pattern you want future pages to inherit by simply using the class. A scroll-reveal observer keyed to a generic class like `.bb-reveal` belongs in `@base`: define the class in base CSS and self-invoke the observer in base JS, and any future page can opt in just by adding the class. **The same logic applies to any class your design uses across multiple sections:** define the rule once under `/* @base */` so every future page on this DS inherits it without redefining. Putting the rule under one `/* @section */` marker leaves the other sections unstyled; putting it under `/* @page */` forces each new page to reauthor it, and small properties like alignment, max-width, or color will drift across pages. The class names are yours to choose based on the design — the rule is about routing shared rules to the layer that gets inherited. Reach for `/* @page */` only when the behavior is genuinely specific to this one page and won't generalize — a one-off countdown, an image lightbox initialized only on a gallery page, page-scoped utility classes, or an observer keyed to selectors no other page would use. When in doubt, prefer `@base`: future pages get it for free, and there's no duplication if another page wants the same effect.

**Creating a design system without a page.** If you are setting up a design system on its own (no associated page), use `create-design-system` and pass the same HTML document shape with `<body></body>` empty. The `<head>` carries everything: `<style>` blocks with `/* @tokens */`, `/* @reset */`, `/* @base */` markers; `<link>` tags for Google Fonts; `<script>` for base JS. The format and conventions are identical to `generate-page` — the only difference is whether `<body>` contains content.

**Whatever shape of spec the user provides** — YAML frontmatter (Google DESIGN.md), free-form markdown with embedded CSS, JSON token files (DTCG / Style Dictionary), plain CSS dumps, or hybrid — interpret its intent and emit our format. The work of mapping author conventions to our token namespace happens here, not at runtime. Apply the canonicalization rules in `core/design-tokens.md` so the structured token map uses `--ds-{category}-{name}` regardless of what the source spec called things.

**Variants.** If the source spec defines multiple values for the same role via runtime selectors (`[data-brand="X"]`, `[data-theme="dark"]`), media queries (`prefers-color-scheme`), or sibling theme files, treat these as alternative brand seeds rather than within-DS variants. A WordPress install is one site; one brand applies. Pick the canonical brand — asking the user which one if ambiguous, defaulting to the parent or first-listed brand if obvious from context — set those values as the `:root` token defaults, and **document the alternative values in `guidance` prose** with their hex codes so the user has a record. Do not emit `[data-X]` selectors or media-query token blocks; the storage layer does not preserve them and will surface a warning if they appear.

### Preserving a user-supplied spec

When a user has supplied a brand spec, design brief, or design system document, the `guidance` field on `create-design-system` has one job: store the spec's prose verbatim. **For that field, you are a transcriber, not an editor.** Copy the user's words unchanged. If you find yourself rewriting, paraphrasing, restructuring, or "cleaning up" anything, stop — that is the wrong behavior. There is no length cap.

The only edits permitted are the four env-translations:

1. **Selectors on `<html>` or `<body>`** describe wrapper-class behavior. Mention them in prose ("the spec applies a wrapper class for X behavior"); do not emit them as selectors.
2. **Multiple brand variants** via `[data-brand="X"]`, media queries, or sibling theme files. Pick the canonical brand for `:root` and document the alternative variants in `guidance` prose with their hex codes (see the Variants section above).
3. **Token names** canonicalize to `--ds-{category}-{name}` in the structured token map. The original author-supplied names (e.g., `--brand`, `--ink`, `--surface`) get mentioned in `guidance` prose so the relationship is documented. Canonicalization applies to the structured tokens only; in `guidance` prose, refer to whatever the spec called things.
4. **Strip YAML frontmatter only.** Code fences flow through unchanged. The Quick-start CSS block, HTML examples like `<p class="pullquote">…</p>`, and exact value blocks like `--brand: #FCB040;` are load-bearing parts of the spec; preserve them.

If something does not fit any of the four translations above, preserve it verbatim. When in doubt, preserve.

**Examples to follow:**

- **Preserve exact values, even if they look unusual.** GOOD: spec says `backdrop-filter: blur(24px)` → guidance contains `backdrop-filter: blur(24px)`. BAD: spec says `backdrop-filter: blur(24px)` → guidance contains `backdrop-filter: blur(12px)` (don't round to a value you "know" as canonical).
- **Preserve standalone subsections — do not consolidate.** GOOD: spec has Hero pill, Pullquote, Sticky CTA, Numbered steps as separate H3s → guidance preserves all four as separate H3s. BAD: spec has four separate H3s → guidance folds them under one "Components" H2.
- **Preserve code fences — they carry load-bearing content.** GOOD: spec has a "Quick-start CSS" code block → guidance preserves the entire block. BAD: spec has a code block → guidance drops it as "redundant."
